# Grafture ‚Äî Copilot Agency Upgrade Plan

_Scope: the Copilot agency feature (Step 6) as deepened by the GF-9 detectors (Step 9). Grounded in a read of the actual source, validated against a checked-in miniature of the HRSA `Health_Center...csv` + OPAIS `OPA_CE_DAILY_PUBLIC.JSON` pair._

---

## 0. What the user wants (the anchor)

Make the Copilot able to derive the _correct_ relational schema from a real heterogeneous pair ‚Äî the HRSA health-center CSV and the OPAIS 340B JSON ‚Äî **without hand-holding**: unnest the JSON's repeating sub-entities, find the foreign keys that actually exist (including the NPI bridge that only shows up once arrays are unnested), and reject the look-alike joins that don't. Today it can only find the single scalar `grantNumber ‚Üî Health Center Number` link and models the JSON as one wide, opaque table. The goal is a schema that matches what a human data engineer derives by hand, produced by the agent's own investigation.

This plan is judged by one regression: on the (miniaturized) HRSA+OPAIS fixture, does the agent reproduce the hand-derived schema? Everything below exists to move that needle. Nothing here adds hosted/paid functionality to `packages/core` or `apps/web` (the licensing line in CLAUDE.md).

## 1. How the agency works today (accurate recap)

Two nested loops:

- **Inner** (`AnthropicBrowserProvider.propose`, `MAX_PREVIEW_ITERATIONS = 3`, `tool_choice: "any"`): the model may call `preview_export`, `inspect_source`, or finalize with `submit_schema_response` (`COPILOT_RESPONSE_TOOL`). Read-only, in-memory. If the budget is spent without finalizing, the provider _forces_ a `submit_schema_response` (`AnthropicBrowserProvider.ts:124-129`).
- **Outer** (`runCopilotLoop`, `DEFAULT_MAX_ITERATIONS = 4`): propose ‚Üí `applyActions` ‚Üí feed rejections back ‚Üí revise. It is a _rejection-correction_ loop (`agentLoop.ts:76-121`).

The model is **not** flying blind on relationships: `buildDynamicContext` (`systemPrompt.ts:287`) injects `<detector_findings>` ‚Äî precomputed output of `detectJoinKeys`, `detectPrimaryKeys`, `detectValueSets`, `detectSemanticTypes`, `detectCompositeKeys`, `detectFunctionalDependencies`, top-N gated by `MAX_JOIN_FINDINGS = 8` (`systemPrompt.ts:94`). This is a strong design and most generic "add a relationship tool" advice is already satisfied by it.

**Provider surface note (load-bearing for P3/P4). [R7-corrected]** The inner tool loop is implemented in **two files, not three** ‚Äî verified: exactly two `MAX_PREVIEW_ITERATIONS` definitions exist, `AnthropicBrowserProvider.ts:36` (tool list :84, `tool_choice: "any"`) and `openaiCompatible.ts:21` (tool list :222, `tool_choice: "required"` :234). There are **three provider _classes_** but only two loops: `OpenAiBrowserProvider` and `LocalBrowserProvider` **both `extends OpenAiCompatibleProvider`** (`OpenAiBrowserProvider.ts:30`, `LocalBrowserProvider.ts:50`), sharing the `openaiCompatible.ts` loop. A tool/iteration change is therefore a **2-file edit** covering all three providers via inheritance. `LocalBrowserProvider`'s JSON-fallback path (`openaiCompatible.ts:268-293`) additionally runs with **no tools** and cannot host a probe. (The earlier "three sites" wording double-counted the inheritance.)

## 2. What is already strong ‚Äî do not rebuild

- The GF-9 detector suite reasons over **values, not names** (Jaccard overlap, inferred grain, format-mismatch/normalize warnings, PK/composite-key/FD/semantic-type detection). This is the core moat and it is well done.
- Prompt-cache split (static instructions cached, dynamic context after the breakpoint) ‚Äî keeps the multi-call loop affordable.
- Prompt-injection guardrails (`<data_handling>`), even-spaced scan sampling (`sampleScanRows` ‚Äî avoids sort bias), and leading-zero-aware range stats (`asRangeNumber`) are all correct and non-obvious. Keep them.

## 3. Root-cause gaps (evidence ‚Üí impact)

### GAP 0 ‚Äî The 1,000-value scan cap bounds _discovery_, not just stats _(prerequisite for A/B)_

`sample.ts:3` `MAX_SCAN_ROWS = 1000`. Parsers call `sampleScanRows(records)` (`json.ts:147`; CSV/XLSX equivalents) so a source retains at most 1000 evenly-sampled rows; `collectDistinctValues` (`sample.ts:125-140`) caps `distinctValues` at ‚â§1000. `detectJoinKeys` compares `field.distinctValues ?? field.samples` (`detect/index.ts:99-101`), so **every** overlap/containment number is computed over ‚â§1000 values per side, regardless of file size. On a 93,018-row vs 18,855-row pair, the _sampled_ intersection of a high-cardinality key badly understates the true overlap. Grain ratios survive sampling; **join discovery does not.** The full-file figures this plan reasons about ("~3,217 shared NPI", "96% containment") are **not reachable** by any detector ‚Äî or by a "live" probe over `distinctValues` ‚Äî until this cap is addressed for discovery. This is a prerequisite, not an optional tuning knob.

### GAP A ‚Äî JSON arrays are stringified, so 1:N structure and array-level joins are invisible _(highest impact)_

`json.ts:59-68`: `flattenValue` handles nested **objects** to `MAX_FLATTEN_DEPTH = 2` (good ‚Äî `streetAddress.city` becomes a column) but **arrays** hit `JSON.stringify(value)` and become a single opaque text column.

Consequence for the flagship use case:

- `npiNumbers`, `medicaidNumbers`, `shippingAddresses`, `contractPharmacies` each collapse to one JSON-string column. The detectors never see their leaf values.
- The **NPI bridge** (CSV `FQHC Site NPI Number` ‚Üî JSON `npiNumbers[].npiNumber`) is undiscoverable ‚Äî the JSON side is a string blob, not a value set.
- The **child-table structure** (one table per array) can never be proposed, because the model never sees these as repeating sub-entities.
- Net: on HRSA+OPAIS the tool can only find the single scalar `grantNumber ‚Üî Health Center Number` link and models the JSON as one wide table ‚Äî materially short of the correct schema.

### GAP B ‚Äî Jaccard under-detects the exact FK shape that matters

`detect/index.ts:831-903::detectJoinKeys` gates on **symmetric Jaccard** ‚â• 0.3 (`DEFAULT_MIN_OVERLAP`, :78/:864). A real foreign key is a _subset_ relationship: the child's keys are contained in the parent's, but the parent has many more ‚Äî high **containment**, low **Jaccard**. `Health Center Number ‚Üí grantNumber` is high one-way containment but low the other way; the FK signal lives in the asymmetry Jaccard averages away, so legit 1:N FKs into a large dimension get filtered at the 0.3 gate. Note the containment _numerator_ is already computed: `sharedValues = intersectionSize(left.full, right.full)` (`detect/index.ts:861`) ‚Äî only the ratio and the gate are missing. (Compounded by GAP 0's sampling.)

### GAP C ‚Äî The agent cannot run a relationship/normalization probe it hypothesizes

Findings are a precomputed global top-N (`MAX_JOIN_FINDINGS = 8`). The only investigative tools are `inspect_source` (one column, ‚â§100 values, and for an array column it returns useless JSON strings) and `preview_export`. The model can form a hypothesis from names/semantics ("these NPI columns should join once I unnest") but has **no tool to verify it** ‚Äî no on-demand overlap test, no "effect of stripping leading zeros," no row-tuple fetch. Its agency is bounded to "look at one more column."

### GAP D ‚Äî Investigation budget is shallow and biased toward early finalize

`MAX_PREVIEW_ITERATIONS = 3` with `submit_schema_response` available from step one nudges the model to finalize before investigating a 56-column CSV + nested JSON. There is no phase whose job is "gather evidence before proposing"; the outer loop only corrects rejected field names.

### GAP E ‚Äî N:M / child-FK modeling is not closed-loop

`planRelationship` (`joinSuggestions.ts:111`) expresses `N:1` as a flipped `1:N`, but there is no auto-junction for `N:M` and, once arrays are unnested (Gap A), nothing auto-proposes the child‚Üíparent 1:N FK. The applied schema can lag the evidence.

### GAP F ‚Äî Grain is inferred from flat source columns, not the modeled schema _(observed in the real run)_

`keyRole`/`inferGrain` (`detect/index.ts:194-223`) decide unique-vs-duplicated from `stats.distinct == stats.nonEmpty` on the **raw source column**. In the denormalized HRSA CSV, `Health Center Number` repeats up to 265√ó (one row per site) ‚Üí reads "duplicated" ‚Üí the `Health Center Number ‚Üî grantNumber` join is reported **N:M**, even though after the org/site split that column is a unique PK and the real grain is **1:N** (organization ‚Üí covered*entity). The detector never re-evaluates grain against the schema being \_proposed*. In the real run this made the copilot dismiss the most important cross-file link as "N:M at best‚Ä¶ not a clean FK." Verified full-file: 96%/42% containment, one grant ‚Üí up to 442 entities, one org = one HCN ‚áí ordinary 1:N.

### GAP G ‚Äî The agent conflates FK-enforceability with relationship _representation_ _(observed in the real run)_

Given partial coverage (42% one way; ~25% unmatched on `siteId ‚Üî BPHC`), the copilot drew **zero** cross-file edges and declared the domains independent ‚Äî a modeling decision made from an _integrity_ fact. Two causes: (1) **doctrine gap** ‚Äî `DESIGN_DOCTRINE`/`ANALYSIS_GUIDANCE` (`systemPrompt.ts:190-213`) never say a partial-coverage relationship is still modeled (nullable/soft FK or bridge table), so "can't enforce" collapses to "don't link"; (2) **workflow gap** ‚Äî nothing in the finalize workflow _requires_ testing cross-source candidates before submitting, so linking is opportunistic (here it happened only on an explicit user re-probe). Also: `detectFormatMismatch` (`detect/index.ts:152-185`) reports _which_ normalizer helps but not _how much_, so the model over-warned ("needs whitespace + case normalization" when that recovered ~39 of ~14,000 matches ‚Äî the real gap is coverage, not format).

## 4. Architecture of the fix

Three seams change; the rest is reused.

1. **Parse seam (`packages/core/src/parse`).** `parseJson` gains the ability to emit a _child_ `Source` per array-of-objects. This changes the parser return contract from `Source` to `Source[]` (see P1 for the cascade). Pure, tested in core.
2. **Detect seam (`packages/core/src/detect`).** `detectJoinKeys` gains directional containment on `JoinKeyCandidate`; a new exported `probeJoin()` helper (core, pure) computes live overlap/containment/grain/format for an arbitrary field pair. The scan cap gets a discovery-specific lift (GAP 0). Pure, tested in core.
3. **Agency seam (`apps/web/src/ai` + `apps/web/src/copilot`).** A new `PROBE_JOIN_TOOL` wraps core's `probeJoin()`; the tool loop's iteration budget and submit-gating change. **All three provider sites** (¬ß1 note) are edited. Modeling logic (junction scaffolding) is web-side apply-plan work calling only the zod action protocol.

**Boundary discipline (CLAUDE.md licensing line):** all _modeling/detection_ logic (containment, `probeJoin`, grain) lands in `packages/core`; only _tool registration and provider wiring_ land in `apps/web`. No detector logic in web; no React/network in core.

## 5. Implementation plan (PR-by-PR, ordered)

Ordering is a hard dependency chain: **PR-0 ‚Üí PR-1 ‚Üí PR-2 ‚Üí PR-3 ‚Üí PR-4 ‚Üí PR-5**. Each PR is independently testable and shippable.

### PR-0 ‚Äî Lift the scan cap for _discovery_ _(fixes Gap 0; prerequisite)_

The detectors need more than 1000 values per column to make full-file overlap claims, but retaining 90k strings per field per source is too heavy. Split the concern:

- Keep `MAX_SCAN_ROWS = 1000` for _display samples_ and _row tuples_ (`sampleRows`, `topValues`) ‚Äî unchanged.
- Add a separate, larger `MAX_JOIN_SCAN` governing **only** the `distinctValues` set that join/containment detection reads. **[R7] Make it adaptive, not a fixed 20,000.** A fixed 20k is still ~21% of the real 93,018-record OPAIS JSON, and sampled containment _understates_ true containment for high-cardinality keys ‚Äî so a fixed cap can still miss the flagship bridge (real NPI ~53% containment, right on PR-2's gate; a 20k window can push it under). Set `MAX_JOIN_SCAN = min(source.rowCount, HARD_CAP)` with `HARD_CAP` a memory ceiling (e.g. 100,000 distinct/column), so files under the ceiling scan in full and only genuinely huge sources degrade. Document the ceiling; do not claim "full-file numbers reachable" above `HARD_CAP`.
- **This is a second, wider row traversal, not a free widening.** Today all parsers call `sampleScanRows` _first_ (`json.ts:147`, `csv.ts:30`, `xlsx.ts:97`) and build every column from the ‚â§1000-row window, so `buildSourceField`‚Üí`collectDistinctValues` (`util.ts:42`, `sample.ts:125`) can never exceed it. All three parsers _do_ retain the full row set in memory before narrowing (`Papa.parse` rows, `sheet_to_json` rows, `extractRecords` records), so a wider pass has data to iterate. Collecting 20k distinct join values requires iterating up to ~`MAX_JOIN_SCAN` source rows for the join columns ‚Äî a wider pass whose _tuples_ are discarded (only the distinct `Set<string>` is kept). Add `collectJoinValues(values)` that streams into a `Set` capped at `MAX_JOIN_SCAN` distinct entries.
- **`buildSourceField` signature change.** `buildSourceField(values, name)` (`util.ts:36`) takes one `string[]` and sets `distinctValues` (`util.ts:42`); a field has no slot for a second, wider set. Widen it to `buildSourceField(values, name, joinValues?: string[])` and have each parser pass `collectJoinValues(widerColumnValues)`. `util.ts` is part of PR-0's edit set.
- **Persistence: strip the wide set explicitly ‚Äî you cannot rely on an "off-schema" field.** `Source` _is_ `SourceSchema` and `ParsedSource = Source` (`parse/types.ts:84`, `parse/index.ts:47`) ‚Äî there is no separate runtime-vs-persisted type, so `joinValues` must be added to `SourceSchema` as optional. And IndexedDB persistence bypasses zod: `saveProjectRecord` (`apps/web/src/persistence/projectStore.ts:62`) `structuredClone`s the raw record to disk _before_ any `safeParse`, so an in-memory field is written regardless of the schema. The 1.1M-string blow-up (20k √ó ~56 columns) therefore happens unless the field is **explicitly deleted before every write**. Mirror the existing `sampleRows` precedent, which is in `SourceSchema` (`types.ts:76`) and stripped by an explicit `delete exported.sampleRows` over a `{...source}` shallow copy in the export path (`serialize.ts:26-27`): add the same for `joinValues` in **both** the file-export path (`serialize.ts`) and the IndexedDB save (`projectStore.ts` / `saveProjectRecord`, the chokepoint all autosave callers funnel through). **Clone before deleting** ‚Äî `saveProjectRecord` receives the live in-memory sources and a debounced autosave fires _during_ the upload session, so a bare `delete record.sources[i].joinValues` would strip the wide set from the running store; write `record.sources.map(s => { const c = {...s}; delete c.joinValues; return c; })` before the `kv.set`.
- **Reload degradation is accepted and must be documented.** After a reload the raw file and wide rows are gone (PR-0 does not retain the extra rows) and `joinValues` was stripped, so it cannot be recomputed without re-upload. Join/containment detection and `probe_join` fall back to `distinctValues` (‚â§1000) in a reloaded session ‚Äî full fidelity only in the upload session; re-uploading the file restores it. This is the deliberate trade against persisting 1.1M strings; state it in the tool description so the degradation is not invisible.
- Files: `parse/sample.ts` (new constant + `collectJoinValues`), `parse/util.ts` (`buildSourceField` 3rd arg), `parse/json.ts`/`csv.ts`/`xlsx.ts` (wider join pass), `parse/types.ts` (optional `joinValues` on `SourceSchema`), `serialize.ts` + `projectStore.ts` (clone-then-`delete joinValues` on write), `detect/index.ts` (only `fieldValues`@:99-101 prefers `joinValues ?? distinctValues ?? samples`). Leave `detectValueSets`@:345 and `inspectSourceTool.ts`@:56 on `distinctValues` intentionally ‚Äî enum detection wants the capped set, inspect is display.
- Tests: `packages/core/test/sample.test.ts` ‚Äî a 30k-row synthetic source yields >1000 distinct join values; display `samples`/`sampleRows` still ‚â§1000. A persistence test asserting a saved record's sources carry NO `joinValues` (both export and IndexedDB paths).
- **If PR-0 is deferred, ¬ß6's acceptance numbers must be restated against a 1000-value sample** ‚Äî the NPI-bridge and Medicare-decoy fixtures below are sized so they pass at either cap, but real-file claims require this PR.

### PR-1 ‚Äî Unnest JSON arrays of objects into child sources _(fixes Gap A)_

In `parse/json.ts`, when a field is an array of objects, emit a **child `Source`** (e.g. `OPA_CE_DAILY_PUBLIC.npiNumbers`) whose fields are the array elements' leaf keys, plus a synthetic parent-link column (`_parentKey`, `type: "text"`, carrying the parent record's detected/first key value).

Concrete contract change and its cascade (deeper than one layer ‚Äî the file‚Üîsource model is 1:1 today):

- `parseJson` returns `Source[]` (parent first, then children) instead of `Source`. `parse/index.ts::parseSource` return type widens to `Source[]`; CSV/XLSX wrap their single `Source` in an array.
- The break is **not** at direct `parseSource` callers (only `readAndParse.ts:17` calls it, itself typed `Source`). It is one layer up, in the web file-intake model:
  - `readAndParse.ts:17` (`readAndParseFile`) return type ‚Üí `Source[]`.
  - `SourcesPanel.tsx:143` does `const source = await readAndParseFile(file); addSource(source)` ‚Äî `addSource` (`schemaStore.ts:492`) wraps one `commitSnapshot`, i.e. one undo step. Add a new `addSources(sources: Source[])` store command that pushes all sources in **one** `commitSnapshot` (CLAUDE.md: one action = one undo step; N children must not be N undo entries). Ids are already assigned by the parser (`resolveMakeId`) and `Source` has no canvas position (only tables are placed, `parse/types.ts`), so there is no layout work ‚Äî just a batched push.
  - `NewProjectModal.tsx:109-110`'s `PreparedFile` models **one source per file** (one `id`, one `file.size`, de-duped by name), and `:170` does `files.map(entry => entry.source)`. Emitting N+1 sources per JSON file breaks this 1:1 model: `PreparedFile.source` becomes `sources: Source[]`, `:170` becomes `files.flatMap(e => e.sources)`, `file.size` attributes to the parent (0 for children), and child names de-dupe via `dedupeNames`.
- **`flattenValue` cannot just branch.** It `JSON.stringify`s all arrays (`json.ts:66`) and `recordFieldValues` keeps only `fieldValues[0]` (`json.ts:132`), discarding the per-record array before fields are built. Child emission needs a **separate record-level traversal** that collects each array's element records ‚Äî a new parser-flow step, not an in-place edit to `flattenValue`. Arrays of scalars keep the existing stringify path.
- `id` allocation for each child via the same `resolveMakeId(opts)` generator; child `name` de-collided with `dedupeNames` against existing source names.
- Child rows are built from the **same wider join pass** (‚â§`MAX_JOIN_SCAN` sampled parents after PR-0) ‚Äî so a child's completeness is bounded by how many parents were scanned; document this.
- Populate child `sampleRows` (project the child's leaf columns into row tuples) so `detectCompositeKeys`/`detectFunctionalDependencies` can run on children ‚Äî they no-op without it (`detect/index.ts:454,581`). Note a child's tuple count is bounded by parents scanned; with ~one array-record per parent a child may fall under `MIN_TUPLE_ROWS=20` and its FD/composite detection silently yields nothing ‚Äî accepted.
- **Parent-link: one surrogate column per side, NOT projected parent leaves.** Two temptations to avoid: (1) guessing "the parent key" at parse time ‚Äî `parseJson` has no PK detection (that runs later, web-side in `detectPrimaryKeys`); (2) projecting _every_ parent scalar leaf into the child as `_parent.<name>` ‚Äî that floods `detectJoinKeys` (which compares all field pairs across all source pairs, `detect/index.ts:848-891`, with no lineage exclusion) with ~overlap-1.0 self-projections that sort to the top and evict the real, low-overlap bridges under `<detector_findings>`'s `slice(0, MAX_JOIN_FINDINGS=8)` (`systemPrompt.ts:108`) ‚Äî burying the very NPI (25% overlap) and grant bridges PR-1 exists to surface.
  Instead: inject a single **synthetic surrogate** ‚Äî a `_rowId` on the parent (its record index) and a matching `_parentId` on each child element ‚Äî and carry the parent‚Üíchild **lineage** explicitly (see below). The surrogate is a _structural_ link, NOT a detector finding.
  **Mark synthetics and exclude them from every detector.** Add `synthetic?: true` to `SourceField` (`SourceSchema`) and guard `detectPrimaryKeys`, `detectJoinKeys`, `detectValueSets`, `detectSemanticTypes`, `detectCompositeKeys`, `detectFunctionalDependencies` to skip synthetic fields. This is required, not cosmetic:
  - `pkNameRank` gives `_rowId` (ends in `id`) **rank 1**, while `grantNumber`/`Health Center Number` (spaces, not `_`) get **rank 3** (`detect/index.ts:225-240`) ‚Äî so an unguarded `_rowId` sorts _above_ the real semantic keys in `detectPrimaryKeys` and would be presented as the parent's top PK, breaking ¬ß6 acceptance #1 and the content-aware moat. Renaming does not help (`__rowId` still ends in `id`); exclusion does.
  - `_rowId` is `0..N`; unguarded it joins any CSV integer column by containment (0..149 ‚äÇ 0..199 = 1.0 ‚â• œÑ, and the >12 distinctness floor does not protect it) ‚Äî a spurious top-ranked "FK" eating a `MAX_JOIN_FINDINGS` slot.
  - (FD/composite already exclude a unique `_rowId` via their `distinct < nonEmpty` gate, `:591`/`:468`; the marker makes it explicit and covers `_parentId` on the child too.)
    The two content bridges need **no** projection and are the detectors' job: `grantNumber ‚Üî CSV Health Center Number` from the **unchanged parent source**, `npiNumbers.npiNumber ‚Üî CSV NPI` from the child's **own leaf**. Unnesting is what makes the NPI leaf visible; that is the whole point.
- **Minimal-diff fallback** if child-sources prove too invasive this cycle: additionally index array-of-object **leaf values** as pseudo-columns (`npiNumbers[].npiNumber`) on the parent source so overlap detection works before full child modeling. This keeps the parser return type as `Source` and is a strictly smaller PR; note it does NOT deliver child-table structure (only PR-1-full does).
- Store lineage on the child `Source`: add optional `derivedFrom?: { parentId, arrayField }` to `SourceSchema` so the web layer can render `derived_from` in `summarizeSources` and emit the structural FK (PR-5). Parent and child both carry the surrogate columns marked `synthetic: true`.
- Files: `parse/json.ts` (child extraction, surrogate injection, lineage), `parse/types.ts` (`synthetic?`, `derivedFrom?` on `SourceSchema`), `parse/index.ts`/`readAndParse.ts`/`SourcesPanel.tsx`/`NewProjectModal.tsx`/`schemaStore.ts` (the `Source[]` cascade above), `detect/index.ts` (skip `synthetic` fields in all six detectors), `copilot/systemPrompt.ts` (render `derived_from`).
- Tests: `packages/core/test/parseJson.test.ts` ‚Äî an array-of-objects input yields N+1 sources; parent `_rowId` + child `_parentId` present and marked `synthetic`; `derivedFrom` set; leaf keys become fields; arrays of scalars stay stringified. `detect` tests: a `synthetic` column is never a PK/join/valueset candidate.

### PR-2 ‚Äî Make join detection containment-aware _(fixes Gap B)_

In `detectJoinKeys`, add `containmentLeft = shared / |left.full|` and `containmentRight = shared / |right.full|` to `JoinKeyCandidate` (reusing the already-computed `sharedValues`, `detect/index.ts:861`). Gate on `max(containmentLeft, containmentRight) ‚â• œÑ` **OR** the existing Jaccard gate, keeping Jaccard secondary. **[R7] œÑ = 0.5 is too aggressive for the flagship bridge ‚Äî start œÑ = 0.4.** The real NPI bridge sits at ~53% containment; with GAP 0's sampling only lowering the observed figure, a 0.5 gate risks suppressing the exact FK PR-1 exists to surface. œÑ = 0.4 keeps margin above incidental matches; the distinctness floor still blocks enums, and `probe_join` (PR-3) is the escape hatch.

**The containment path needs its OWN guard ‚Äî `minSharedValues` does not stop enum false-positives.** A genuine 3-value enum `{A,B,C}` fully present in a large column has `sharedValues = 3` (clears `minShared=2`) _and_ `containmentLeft = 3/3 = 1.0 ‚â• œÑ` ‚Äî it would surface. The containment gate is exactly what re-admits the incidental low-cardinality match. Guard it with a **distinctness floor on the contained (smaller) side**: only let the containment path fire when the smaller side has `> DEFAULT_MAX_DISTINCT` (12) distinct values ‚Äî i.e. it is not itself a closed value-set / enum (the same cardinality `detectValueSets` uses at `detect/index.ts:313`). `minSharedValues` stays as the _Jaccard-path_ guard, unchanged.

- **Known limitation (documented, not asserted away):** a legitimate FK into a _small_ dimension (‚â§12 distinct parent keys ‚Äî a compact lookup table) is cardinality-indistinguishable from an enum, so the floor suppresses it on the containment path; it must be found via the Jaccard path instead. That path is adequate for small dimensions (a ‚â§12-value child ‚äÇ ‚â§12-value parent has high Jaccard); the false-negative only bites when a ‚â§12-distinct column joins a _large_ column, which is genuinely ambiguous. State this trade in the finding, don't hide it.
  Surface both containment directions in `JoinKeyCandidate` and in `<detector_findings>` so the model and reranker can read "left ‚äÜ right ‚áí left is the FK side."
- Files: `detect/index.ts` (type + gate + distinctness floor), `copilot/systemPrompt.ts` (render containment in the findings block).
- Tests: extend `packages/core/test/detect.test.ts` ‚Äî an FK-shaped pair (child ‚äÇ large parent, Jaccard < 0.3, contained side > 12 distinct) now surfaces; a 3-value enum fully contained in a big column does NOT surface (distinctness floor). The existing threshold test at `detect.test.ts:139-143` continues to reject its pair (it fails _both_ gates: `sharedValues=1 < 2` on the Jaccard path and `max(containment)=0.25 < œÑ` on the containment path) ‚Äî verify this explicitly rather than assuming `minShared` alone carries it.

### PR-3 ‚Äî Add an on-demand `probe_join` tool _(fixes Gap C)_

Optionally export a new `fetchRowTuples(source, n)` for FD/coincidence checks (distinct from the existing `sampleScanRows` helper and the `Source.sampleRows` field ‚Äî do not reuse either name).
Web: new `PROBE_JOIN_TOOL` (`apps/web/src/copilot/probeJoinTool.ts`) that validates input and calls the core helper. Register it in the tool list at the **two** loop sites (`AnthropicBrowserProvider.ts:84`, `openaiCompatible.ts:222`) ‚Äî the latter covers OpenAI _and_ Local via inheritance (¬ß1 R7 note), so no third edit exists; the no-tools Local JSON-fallback path (`openaiCompatible.ts:268-293`) is documented as unsupported for probing.

- Tests: `packages/core/test/probeJoin.test.ts` (core helper); a web-side test asserting the tool dispatches to the helper.

### PR-4 ‚Äî Split investigation from finalization _(fixes Gap D)_

Raise `MAX_PREVIEW_ITERATIONS` to 5‚Äì6 and **withhold `submit_schema_response`** for the first N inner rounds (offer only probe/inspect/preview), enabling it once an evidence budget is spent or the model signals ready.

- **This is a control-flow change, not a threshold tweak.** Both loops build `tools` **once before** the loop (`AnthropicBrowserProvider.ts:84`, `openaiCompatible.ts:222`). Withholding submit for the first N rounds requires constructing the tool list _inside_ the loop, keyed on `iteration` (submit present iff `iteration >= N` or the model signalled ready). Refactor at both sites.
- The forced-finalize path (`AnthropicBrowserProvider.ts:124-129`) already builds its own submit-only list ‚Äî unchanged; it still resolves the turn on budget exhaustion, so withholding submit never dead-ends the loop (probe/inspect/preview stay callable throughout).
- Tool-choice differs per provider: Anthropic uses `tool_choice: "any"`, the OpenAI-compatible path uses `"required"` (`openaiCompatible.ts:234`). Keep each as-is; the withhold logic is orthogonal to which forcing value is used.
- Tests: provider-level test asserting `submit_schema_response` is absent from the tool list for the first N rounds and present after, at both loop sites.

### PR-5 ‚Äî Close the modeling loop for N:M and child FKs _(fixes Gap E)_

**There is no shared "apply-plan builder" both paths use ‚Äî pick the path deliberately.** `buildApplyPlan` (`joinSuggestions.ts:132`) is called **only** from the suggestion-panel accept flow (`useSuggestions.ts:106`); the copilot's actions come entirely from the LLM's `submit_schema_response` (`responseTool.ts` ‚Üí `agentLoop.ts:89`) and never touch `buildApplyPlan`. So "closing the loop" splits by surface:

- **Copilot surface (primary, matches ¬ß0's "agent's own investigation"):** this is _prompt_ work, not a builder. **The copilot has no lineage channel today ‚Äî add one.** `summarizeSources` (`systemPrompt.ts:63-96`) renders only name/type/samples/stats; nothing tells the model that child C was unnested from parent P's array. And copilot actions come entirely from the LLM's `submit_schema_response` (never `buildApplyPlan`), so "structural emission" cannot mean a deterministic builder on the copilot path ‚Äî it means the model must be _told_ the lineage and instructed to emit the FK. So: extend `summarizeSources` to render, per child source, a `derived_from: {parent, arrayField}` and `link: "_parentId ‚Üí _rowId"` line. This is the structural (non-overlap) channel; the surrogate columns are excluded from `detectJoinKeys` (PR-1), so the link is never rediscovered by value overlap.

Then in `systemPrompt.ts`, instruct the model: when a detected grain is `N:M`, emit a junction table (one `add_table`, two `add_field` for the FK columns, two `add_relationship` both 1:N, and ‚Äî since `set_pk` sets one field's boolean (`actions.ts:77`) ‚Äî a `set_pk` per composite-key column) rather than a direct N:M; when a source has a `derived_from` lineage, emit the child‚Üíparent 1:N FK on its surrogate pair (`child._parentId ‚Üí parent._rowId`). On the **secondary** suggestion path only (`buildApplyPlan`), this scaffolding can be deterministic. The zod ops all exist (`add_table`, `add_field`, `add_relationship`, `set_pk`, `set_cardinality` ‚Äî verified); actions flow through `applyActions`, invalid ones surfaced, never dropped. No new action op is needed.

- **Suggestion surface (secondary, deterministic):** extend `buildApplyPlan` (`joinSuggestions.ts:132`) so the _accept_ path also scaffolds the junction/child-FK for the non-copilot flow. This is deterministic UI, explicitly not "the agent deciding."
- Tests: (copilot) `apps/web` loop test with a stubbed `ProposeFn` that returns an N:M-grain proposal, asserting the store applies a junction table + two 1:N FKs; (suggestion) a `buildApplyPlan` unit test asserting the same action batch for an N:M candidate. Consistent with `runCopilotLoop`'s injected-effects testability.

### PR-6 ‚Äî Schema-aware grain _(fixes Gap F)_

Grain must be judged against the **modeled entity**, not the flat column. Add an optional `keyContext` to grain inference: a field that is (or is proposed as) a table's primary key is treated as the "one" side regardless of how often its value repeats in the denormalized source.

- Detector-time: when `detectPrimaryKeys` (`detect/index.ts:247`) flags a PK candidate, `inferGrain` consults that verdict for the same field so `detectJoinKeys` reports 1:N (not N:M) for an org-key ‚Üî fact-key pair.
- Copilot-time: `probeJoin` (PR-3) accepts the current schema and, if a probed column is already a PK on the canvas, resolves that side to "one" ‚Äî turning the real run's "N:M at best" into "1:N, nullable FK."
- Files: `detect/index.ts` (`keyRole`/`inferGrain` gain a PK-aware override; `detectJoinKeys` threads PK candidates), `copilot/probeJoinTool.ts` (pass schema through). Pure, core-tested.
- Tests: `packages/core/test/detect.test.ts` ‚Äî a duplicated-in-flat-file column that is a PK candidate yields grain 1:N against a duplicated fact column; regression pinning the HRSA-shaped case.

### PR-7 ‚Äî Linking doctrine + required cross-source phase + relationship classifier _(fixes Gap G)_

Highest-leverage, lowest-risk change ‚Äî pure prompt/helper work, shippable independently of the parser PRs.

- **Doctrine (`systemPrompt.ts` `DESIGN_DOCTRINE`):** add a rule ‚Äî _enforceability and representation are separate. A relationship supported by the data is always represented; partial coverage is modeled as a **nullable/soft FK** (or a bridge table for N:M), never dropped. Never leave two sources as disconnected islands when a cross-source key exists._
- **Workflow (`systemPrompt.ts` `<workflow>` + `ANALYSIS_GUIDANCE`):** make cross-source linking a **required finalize step** for multi-source schemas ‚Äî before `submit_schema_response`, the agent must test each cross-source join candidate (via `<detector_findings>` and `probe_join`) and either represent the survivors or state in `reply` why each was rejected. Mandatory, not opportunistic.
- **Relationship classifier (core helper, `detect/index.ts`):** `classifyRelationship({containmentLeft, containmentRight, grain, nullRate, formatMismatch})` ‚Üí `{ verdict: "enforced_fk" | "nullable_fk" | "not_valid_fk" | "junction" | "no_link", reason }`. Turns raw probe numbers into a consistent modeling decision instead of the ad-hoc reasoning that produced the N:M mislabel + over-warning. Surface the verdict in `<detector_findings>` and as a `probe_join` output field.
- **Quantify normalization gain (`detectFormatMismatch`, `detect/index.ts:152-185`):** report the _marginal_ shared-value gain per normalizer (e.g. `+39` for trim+case) so the model warns proportionally and doesn't call a coverage gap a formatting problem.
- Files: `copilot/systemPrompt.ts` (doctrine + workflow), `detect/index.ts` (`classifyRelationship`, marginal-gain in `FormatMismatch`), `copilot/probeJoinTool.ts` + `<detector_findings>` (surface the verdict).
- Tests: `packages/core/test/detect.test.ts` ‚Äî classifier returns `nullable_fk` for the 1:N partial-coverage HRSA pair and `no_link` for a near-zero-overlap decoy; format-mismatch reports the marginal gain. `apps/web` loop test: a stubbed multi-source proposal that omits an available cross-source FK is driven to add a nullable FK (workflow enforcement).
